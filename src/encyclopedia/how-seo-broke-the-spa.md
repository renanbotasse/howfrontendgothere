
A single-page application built with React or Angular returns an HTML file that looks roughly like this when you view the page source:

```html
<!DOCTYPE html>
<html>
  <head>
    <title>My App</title>
  </head>
  <body>
    <div id="root"></div>
    <script src="/bundle.js"></script>
  </body>
</html>
```

That's it. No content. A search engine crawling this page sees an empty div.

For applications where users need to log in before seeing anything meaningful — dashboards, SaaS tools, internal apps — this doesn't matter. Nobody is searching for your company's invoice management interface. But for any application with publicly discoverable content — product pages, blog posts, news articles, documentation — sending search engine crawlers an empty page is a serious problem.

This tension between client-side rendering and discoverability is one of the main forces that brought server-side rendering back.

---

## How search crawlers work

A search engine crawler downloads a web page's HTML and parses its content. The words, headings, links, and metadata it finds determine how the page is indexed and what searches it appears in. The standard crawler behavior is fast and stateless: download the HTML, extract the content, follow the links.

JavaScript execution is slower. Running a JavaScript engine, waiting for the bundle to download and execute, waiting for data fetching to complete before the UI renders — this adds seconds to crawling a single page. A crawler indexing millions of pages can't afford that overhead for every page.

Google has the most sophisticated crawler. Googlebot has been able to execute JavaScript in some form since 2014, using a headless Chrome instance to render pages. But even Google's JavaScript rendering is not equal to static HTML. There can be delays in indexing, depth limits on crawled pages, and inconsistencies in how dynamic content is handled.

Bing, DuckDuckGo, and most other search crawlers are less capable of executing JavaScript. For sites that care about search visibility across multiple engines, sending an empty page to crawlers is genuinely costly.

## The social media problem

Search crawlers aren't the only audience for server-generated HTML. When you share a URL on Twitter, LinkedIn, or Slack, those platforms fetch the URL and look for OpenGraph meta tags to generate a preview — title, description, image.

```html
<meta property="og:title" content="How SPAs Broke SEO" />
<meta property="og:description" content="The rendering problem that brought server-side back." />
<meta property="og:image" content="https://example.com/preview.jpg" />
```

Twitter's card scraper doesn't execute JavaScript. Slack's unfurl doesn't execute JavaScript. If these meta tags are generated by JavaScript after the page loads, the preview is either empty or shows generic defaults.

An e-commerce product page that shares without a preview image and title loses a meaningful number of clicks. For content-driven businesses, rich social previews are real traffic.

## Workarounds that were tried

Teams used various approaches to solve the crawlability problem without moving to server-side rendering.

**Prerendering services** like prerender.io intercepted crawler requests, rendered the JavaScript in a headless browser, and returned the static HTML. User requests got the SPA; crawler requests got the pre-rendered HTML. This worked but added infrastructure complexity, cost, and a layer that had to be kept in sync.

**Snapshot deployment** meant running the SPA through a headless browser at build time and saving the HTML output for each route. Netlify and Gatsby popularized this for static sites. It worked for sites where the content was known at build time, but not for dynamic content that changed frequently or varied by user.

**Meta tag injection** solved the social sharing problem specifically: a server component or CDN function checked the user agent, and for known crawlers and social bots, returned a minimal HTML response with the right meta tags even if the full content required JavaScript.

Each workaround was a workaround. They addressed the symptom without addressing the root cause: generating HTML on the client when the crawling infrastructure expected HTML from the server.

---

## Google's evolving position

Google's official advice was "we render JavaScript, don't worry about it." Their crawler documentation described a two-wave indexing process: a first wave that indexed static HTML immediately, and a second wave that re-crawled and rendered JavaScript after a delay. The delay could be days or weeks.

That delay matters for freshly published content. A news site that publishes a JavaScript-rendered article might not have it indexed for days. For content that's time-sensitive, that's not acceptable.

Google's Core Web Vitals update in 2021, which made page experience metrics a ranking signal, added another dimension. SPAs with poor initial load performance — which is most SPAs, because of the JavaScript parsing cost — were penalized in search rankings relative to server-rendered pages with better Largest Contentful Paint scores.

This gave SEO a direct line to engineering priorities in ways it hadn't had before. "We're ranking lower because our SPA is slow" was a conversation that got engineering resources.

---

## The rendering spectrum

The response from frameworks was to make multiple rendering strategies available and composable. Next.js, starting around version 9-10, introduced the concept of choosing a rendering strategy per page:

- Static site generation: HTML built at deploy time, same for all users
- Server-side rendering: HTML generated on each request
- Client-side rendering: HTML is an empty shell, content loads in the browser
- Incremental static regeneration: static HTML rebuilt in the background periodically

The right choice depended on the content. A blog post that rarely changed was a candidate for static generation. A product page that changed every few hours was a candidate for incremental static regeneration. A search results page that depended on the query was a candidate for server-side rendering. A user dashboard that required authentication was fine with client-side rendering.

This granularity was a direct response to the SEO problem. The industry stopped treating rendering strategy as a one-size-fits-all architectural choice and started treating it as a per-route decision based on the content's characteristics.
