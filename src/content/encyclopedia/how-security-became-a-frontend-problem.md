
Security used to be primarily a backend concern. The server validated inputs, protected the database, managed authentication, and enforced access control. The frontend was a presentation layer. What could go wrong in a presentation layer?

Quite a lot, it turns out. The browser is an execution environment for code from multiple sources, your code, third-party scripts, user-generated content, ad networks, analytics, chat widgets. Any of that code can interact with the DOM, read cookies, make network requests, and access browser APIs. When a page runs untrusted code, the consequences are the attacker's imagination.

---

## Cross-Site Scripting (XSS)

XSS is the most common web vulnerability. It happens when an application takes user-supplied content and renders it as HTML without sanitizing it. If a user can submit a comment that contains `<script>alert('XSS')</script>` and your application renders that as HTML, the script executes in the browser of every user who views the comment.

A malicious script can do anything your own JavaScript can do: read cookies, steal localStorage, redirect users, make authenticated API calls, modify the DOM to add fake login forms.

Modern frameworks defend against XSS by default through escaping. React renders string values by escaping HTML entities before they touch the DOM, `<script>` becomes `&lt;script&gt;`, which displays as text instead of executing as HTML. Angular does the same. The danger zone is when you deliberately bypass this protection:

```jsx
// Safe, React escapes the string
<div>{userComment}</div>

// Dangerous, explicitly opted out of escaping
<div dangerouslySetInnerHTML={{ __html: userComment }} />
```

`dangerouslySetInnerHTML` exists for legitimate use cases, rendering Markdown-to-HTML output, embedding content from a trusted CMS. Used with untrusted user input, it's a direct XSS vulnerability. The name is intentionally alarming.

Content Security Policy (CSP) is the defense layer that limits what scripts can execute even if XSS does occur. A strict CSP can prevent inline scripts from running and restrict which external scripts can be loaded, significantly limiting what an attacker can do with an XSS vulnerability.

## Cross-Site Request Forgery (CSRF)

CSRF exploits the browser's automatic cookie sending. When your browser visits `evil.com`, that page can make a request to `yourbank.com` and the browser will automatically attach your `yourbank.com` cookies, because that's how cookies work. If you're logged into yourbank.com, the request arrives authenticated.

```html
<!-- On evil.com -->
<img src="https://yourbank.com/transfer?amount=1000&to=attacker" />
```

The defense is CSRF tokens: a random value generated by the server, embedded in forms, required on any state-changing request. `evil.com` can make the request but can't include the CSRF token because it can't read it, the same-origin policy prevents cross-origin JavaScript from reading responses from other origins.

The `SameSite` cookie attribute is a newer and simpler defense. A cookie with `SameSite=Strict` is not sent on cross-site requests at all. `SameSite=Lax` allows the cookie on top-level navigations but not on subresource requests like images. Modern browsers default to `Lax` for cookies that don't specify `SameSite`, which mitigates most CSRF attacks without any application code change.

## Third-Party Scripts and the Supply Chain

A modern web page typically loads JavaScript from many sources: Google Analytics, Intercom, Segment, Hotjar, advertising scripts, A/B testing tools. Each of these scripts runs in the same browser context as your application code. Each has access to your DOM, your localStorage, your cookies.

This is the supply chain attack surface. If an attacker compromises one of these third-party services, or if a malicious package finds its way into your dependency tree, they can inject code into your application at scale.

The most significant recent example in frontend: the polyfill.io supply chain attack in 2024, where a service that provided browser polyfills was acquired and the new owners injected malicious code that affected hundreds of thousands of websites.

Subresource Integrity (SRI) is a mitigation for third-party scripts loaded from CDNs. You include a cryptographic hash of the script content in the script tag:

```html
<script src="https://cdn.example.com/library.js"
  integrity="sha384-abc123..."
  crossorigin="anonymous"></script>
```

The browser verifies the hash before executing the script. If the script has been modified, the hash doesn't match and the script doesn't run.

For npm dependencies, auditing the dependency tree (`npm audit`), locking dependency versions, and reviewing what packages have access to is hygiene that most teams practice inconsistently.

## Clickjacking

Clickjacking embeds your page in an `<iframe>` on a different site and tricks users into clicking buttons they can't see. The attacker's site shows a game where you click to score points; invisible under the cursor is a button on your page, "Confirm bank transfer."

The defense is the `X-Frame-Options` header, which tells browsers not to render the page inside an iframe: `X-Frame-Options: DENY`. The `Content-Security-Policy: frame-ancestors 'none'` directive does the same and is more flexible.

## The browser as execution environment

The root of why frontend security is hard is that the browser executes code from sources with different trust levels. Your application code, third-party scripts, user-generated content, and injected code through XSS all run in the same process with access to the same APIs.

Security in the browser is about compartmentalization: limiting what each piece of code can access and do. CSP limits script execution. Cookie attributes limit what's sent where. CORS limits which origins can make API calls. SRI limits which CDN scripts run. None of these defenses is complete, but together they significantly reduce the attack surface.

Frontend developers who understand these mechanisms write more secure applications. The alternative, treating security as purely a backend concern, is how companies end up in breach post-mortems explaining how a third-party chat widget stole user data.
